hexo使用记录

May 7, 2020 This post has been read by person

之前一直将个人主页用Wordpress搭在自己的VPS上，但由于该VPS服务商规模比较小，线路不稳定而且还担心跑路。想着个人站点这种需要长久维护的项目还是需要放置在一个比较稳定的地方的，因此就转移到了Github Page。

用流行的hexo项目 作为网站框架，由于Github Page只能解析静态文件并渲染成页面，因此在推送到Github Page项目之前本地需要有一个转化的过程（即hexo g）。

下载安装node.js之后（注意勾选安装选项中的npm工具），即可使用包管理工具npm来安装hexo框架了，可在任意位置使用命令行执行对应命令：npm install -g hexo-cli，-g意味着全局安装。

接着，在任意空文件夹下初始化一个hexo项目，hexo init，完成后你会看到该文件夹中出现已经构建好了的hexo项目，官方文档要求继续在该文件夹下执行npm install命令以安装相关的依赖包，但查看日志会发现实际在初始化的过程中就已经完成了依赖包的安装。

之后，在scaffolds文件夹中设置post，page，draft框架，添加进去每一个新建的实例都需要的东西，实际上也可以保持默认，这只是为了简化后续的重复劳动。

md文件中开头---内的数据称之为Front-matter，是用来标记该文件的一些属性，下表会对其做些说明：

关键字 作用 示例格式
layout 表明该文本的布局 layout: post
title 该文本的标题 title: hexo使用记录
date 创建日期 date: 2020-05-07 08:00:00
updated 更新日期 updated: 2020-05-07 08:00:00
comments 能否评论 comments: true
tags 标签 tags:<\br> - tool
categories 分类 categories:<\br> - [Sports, Baseball] (此表明归属为Sports的子类Baseball)
excerpt 摘要 hexo更推荐在文本中添加<!-- more -->，此行前的文本将会被作为摘要

进入themes文件夹中能看到默认的landscap主题，同样也可以下载其他主题到此处，hexo主题 。

hexo项目中的-config.yml文件用于配置，同时主题中也存在一个-config.yml，这里的配置方法由于不同的设计将会各有不同，大部分的配置项其名称即表述了其功能，而另外的一些最好的理解办法就是修改后生成本地看其效果。

hexo项目-config.yml：用于主题配置和Github Page仓库配置以及视频播放器的部分

可以随时使用hexo g命令来生成静态网站，hexo项目下会生成public文件夹，并使用hexo s来开启本地服务器预览，如果出现问题，可使用hexo clean来清除生成的网站，然后再次生成。

如果下载的主题中带有除home和archive外的其他标签页，那么就需要使用hexo new <layout> <name>先生成这些页面，注意在主题的layout文件夹中对照准确的布局名。hexo new post <name>同样用来生成文章，这会出现在hexo项目文件夹的\source\_posts目录下，找到同名的md文件并采用markdown标记编写它，生成前保存即可，注意文件名尽可能短，内部的title长度则无所谓。

当一切合心意之后，就可以部署到GitHub Page中，注意配置好hexo项目-config.yml中的Deployment项，然后使用npm install hexo-deployer-git --save安装部署工具，再执行hexo d命令推送至Github Page仓库。

需要使用单独域名时，先在域名服务商处添加两条记录类型为CNAME，主机记录为@和WWW，记录值为Github Page地址的解析记录，等待缓存更新后，再在Github Page的仓库设置中添加Custom domain并保存，该设置会在仓库根目录下生成一个CNAME文件。此操作需要一段时间用于Github生成https证书。

这并不是一个可以详述的过程，不同于wordpress，hexo项目需要一些探索，即便是有主题和插件的存在，也需要依据不同的个性化需求来做一些嵌入和配置，而不像wordpress全程基本可视化。hexo的使用过程难免会有一些代码的配置工作，但这并不是要求先学会这些语法才能操作，项目短小和模块化即使是对于我这种几乎从未完整学习过前端的人来说也十分友好，拆开这些几十行逻辑简单的ejs文件，很容易便能得到期望的样子，当然，always backup。
